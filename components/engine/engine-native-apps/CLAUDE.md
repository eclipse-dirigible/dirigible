# engine-native-apps

User projects can declare a `*.native-app` JSON file that turns an external web server — local OS process or remote HTTP(S) endpoint — into a first-class Dirigible artefact, reverse-proxied under `/services/native-apps-proxy/v1/<basePath>/...` with optional Dirigible-managed authentication and role-based access. The whole feature lives in this module (`org.eclipse.dirigible.components.engine.nativeapps`); the management surface lives in `org.eclipse.dirigible.components.engine.nativeapps.endpoint`.

- **Artefact type string `native-app`, file extension `.native-app`, JPA table `DIRIGIBLE_NATIVE_APPS`.** Synchronizer order is `SynchronizersOrder.NATIVE_APP = 440` (after `PROXY = 430`, before `OPENAPI = 510`). `NativeAppSynchronizer` extends `BaseSynchronizer` (single-tenant — apps are platform-global). The table carries two unique constraints: `UK_DIRIGIBLE_NATIVE_APPS_NAME` (on `ARTEFACT_NAME`) and `UK_DIRIGIBLE_NATIVE_APPS_BASE_PATH` (on `NATIVE_APP_BASE_PATH`) — the registry maps each basePath to exactly one app, duplicates would silently overwrite without the DB-level constraint.
- **Two kinds: `local` and `remote`.** Remote = third-party HTTP(S) URL Dirigible doesn't own; the artefact only declares `config.url`. Local = OS process Dirigible spawns and supervises. Local apps carry a `config.lifecycle.start.commands[]` and optional `config.lifecycle.stop.commands[]`, each entry tagged with `os` ∈ `mac`/`linux`/`windows`. `OsCommandResolver.currentOs()` returns a typed `Os` enum (`MAC` / `LINUX` / `WINDOWS` / `UNSUPPORTED`); `pickForCurrentOs` parses each `command.os` JSON token through `Os.fromToken` (case-insensitive via `Locale.ROOT`).
- **Start modes (local only): `lazy` vs `always`.** ALWAYS apps are spawned by `NativeAppBootstrap` on `ApplicationReadyEvent` and kept alive by `NativeAppMonitorJob` — a `SystemJob` (i.e. `DirigibleJob` subclass) auto-scheduled by the platform's `JobsInitializer` because the class is a Spring `@Component` extending `SystemJob`. Interval is configurable via `DIRIGIBLE_NATIVE_APP_MONITOR_INTERVAL_SECONDS` (default `30`). The job iterates LOCAL apps in ALWAYS mode and restarts dead ones. LAZY apps start on the first proxy request via `LazyStartFilter.startAndAwaitReady(app)`. The synchronizer's `START` phase only acts on ALWAYS apps — LAZY must not be pre-started.
- **Port resolution: prefer-then-allocate.** `PortResolver.resolve(preferredPort)` first probes the declared `config.defaultPort` (loopback `ServerSocket` open/close). On `BindException` (or if `defaultPort` is null) it allocates an OS-chosen ephemeral port via `ServerSocket(0)`. The resolved port is exported to the child as `DIRIGIBLE_NATIVE_APP_PORT` — **the spawned process must read that env var** rather than hardcode a port. See [`dirigiblelabs/sample-library-local-native-app`](https://github.com/dirigiblelabs/sample-library-local-native-app)'s `src/config.ts` for the canonical pattern (`env.DIRIGIBLE_NATIVE_APP_PORT ?? env.PORT`).
- **Readiness probe.** After spawn, `NativeAppProcessManager.awaitReady` polls `isLoopbackPortOpen(port)` every 200 ms up to `DirigibleConfig.NATIVE_APP_READY_TIMEOUT_MS` (default 30 s). Bump it via that enum if the spawn needs first-time `npm install` etc — `SampleLibraryLocalNativeAppIT` sets it to 5 min for that reason.
- **Working directory: `command.dir` resolves under the artefact's project root.** Base is `/registry/public/<project>/`, derived from `app.getLocation()` (first path segment). Null / blank / `"."` `dir` means "the project root itself" — which is what you want when `package.json` lives at the project root. Otherwise `dir` is joined as a subpath. **Do not embed the project folder name in `command.dir`** — the platform already knows it, and embedding it makes the artefact brittle to repo renames, forks, or clones-under-a-different-name (the original symptom was a silent fallback to registry root, where `npm install` failed for lack of `package.json`, the process exited in <1 s and `awaitReady` threw "exited before it became ready"). If a configured subpath doesn't exist on disk, `resolveWorkingDir` logs a WARN with a concrete remediation hint and falls back to the project root rather than registry root.
- **Premature-exit diagnostics.** When the spawned process dies before its port opens, `awaitReady` throws `IllegalStateException` whose **message includes**: PID, exit code, elapsed runtime in ms, resolved port, working directory, and the last ~30 stderr lines captured by `ProcessLogPump` via `RuntimeState`'s ring buffer (`recordStderrLine`/`snapshotRecentStderr`). Operators don't have to chase the per-app child loggers (`org.eclipse.dirigible.nativeapps.<name>.stderr`) to learn what npm/node/python actually printed before exiting — but the message still names those loggers as the source of the full live stream. The ready-timeout path emits a similarly rich message naming `DIRIGIBLE_NATIVE_APP_READY_TIMEOUT_MS` as the knob to turn for legitimate cold starts.
- **Stop / teardown — three layered guarantees in `NativeAppProcessManager.stop`:**
  1. **Author's `lifecycle.stop` command first** (best-effort). Runs in the same `command.dir` as `start` (same project-root-relative resolution), with `DIRIGIBLE_NATIVE_APP_PORT` exported on the subprocess env **symmetric to start** — the stop script needs the resolved port too. Skipped with an INFO line if the **project root** is gone (the workbench's `Unpublish` deletes the project tree before the synchronizer's DELETE fires; running `npm stop` from registry root just produces noisy errors). Skipping is safe — `Process.destroy()` next handles termination.
  2. **`Process.destroy()` on the held root**, with a `STOP_GRACE_MS = 5000` `waitFor`, then `destroyForcibly()` if needed.
  3. **`Process.toHandle().descendants()` walk** captured *before* `destroy()` (the OS reparents orphans to init/launchd once the root dies, after which they are no longer reachable as descendants of the held handle). Any descendant still alive after the grace period is `destroyForcibly()`d, each kill logged at INFO with PID + command. This handles shell-based launchers like `sh → npm → sh → node` where SIGTERM to the root doesn't propagate (especially when `exec` rewrites the held PID into npm, which then forks a fresh sh+node pair that survives npm's exit). The same descendants walk runs in `stopAll()` on JVM shutdown.
- **Synchronizer DELETE has two paths into `processManager.stop` — both must rehydrate config.** `completeImpl(DELETE)` is the normal path (artefact still tracked); `cleanupImpl` is the orphan path (DB row whose source file is already gone). The DB-loaded entity carries `configJson` but not the transient typed `NativeAppConfig` tree, so `runStopCommandsBestEffort` would otherwise bail with "typed config is unavailable" and the stop command would never run. Both callers invoke `NativeAppParser.rehydrateConfig(app)` first. If you add a third teardown path, mirror this.
- **Lifecycle logging is PID + port everywhere.** Every transition (`Starting native app … on port [N] …`, `Native app [foo] spawned: PID [P]`, `Native app [foo] is ready: PID [P], port [N]`, `Stopping native app [foo]: PID [P], port [N]`, `Killing leftover descendant of native app [foo]: PID [d-pid], command […]`, `Native app [foo] (PID [P]) did not stop within … forcibly destroying.`) carries both so operators can cross-reference `lsof -ti tcp:<port>` / `ps -p <pid>` / `pstree <pid>`. Note: `lsof -ti tcp:<port>` returns every PID holding a TCP FD for that port — listeners *and* connected clients. After a proxy round-trip the Dirigible JVM itself shows up there (its outbound HttpClient keep-alive entry in CLOSED state, until the connection-pool reaper sweeps it). Diagnose with `lsof -iTCP:<port> -sTCP:LISTEN -P -n` for the unambiguous listener check.
- **`lifecycle.stop` subprocess env: `DIRIGIBLE_NATIVE_APP_PORT` AND `DIRIGIBLE_NATIVE_APP_PID`.** Both are exported by `NativeAppProcessManager.runStopCommandsBestEffort`. The PID is the held root PID. **Hard rule for stop scripts: target `$DIRIGIBLE_NATIVE_APP_PID`, never `lsof | xargs kill` by port.** `lsof -ti tcp:<port>` returns every PID with a TCP FD on the port, listeners *and* keep-alive clients — so a port-based kill from an author's stop script reliably kills the Dirigible JVM (Spring Cloud Gateway's HttpClient keeps idle connections to upstream ports after every proxy round-trip; the user-reported repro: switch to `mode: always` and republish → synchronizer UPDATE → stop runs `lsof | xargs kill` → Spring shutdown hooks fire). Most apps don't need a custom stop script at all — the platform's `Process.destroy()` SIGTERMs the held PID, and a well-behaved app handles SIGTERM gracefully (Node does by default).
- **Proxy chain — Spring Cloud Gateway WebMvc, modelled on `engine-proxy`.** `NativeAppProxyRouterConfig` registers `/services/native-apps-proxy/v1/**` with the filter order:
    1. `rewritePath` strips the absolute base.
    2. `NativeAppLookupFilter` resolves `basePath → NativeApp`, rewrites the path to the upstream-relative form, attaches the app as a request attribute. Supports an empty-`basePath` catch-all that matches when no named app catches first.
    3. `ExposedPathFilter` enforces `security.exposedPaths` (404 on whitelist miss) and `scopes` (403 on miss). **Important:** native-app scope semantics are intentionally strict — DEVELOPER/ADMINISTRATOR super-roles do NOT grant implicit access. Authors define their app's audience explicitly. If `scopes` is empty / `security` is null, any authenticated caller passes (auth itself is enforced upstream by Spring Security on `/services/**`).
    4. `LazyStartFilter` spawns LOCAL+LAZY apps on demand.
    5. `NativeAppDispatcher` sets the downstream scheme/host/port: LOCAL → `http://127.0.0.1:<resolvedPort>`, REMOTE → `app.remoteUrl`. The current request path (already rewritten in step 2) is reused by SCG's `http()` handler.
    6. `AuthInjectionFilter` consults `AuthenticationInjectorRegistry` and lets the matching `AuthenticationInjector` mutate the outbound request. Lookup is case-insensitive on the type token (`Locale.ROOT` lowercase fold). `BasicAuthenticationInjector` uses `.headers(h -> h.set(AUTHORIZATION, ...))` because `builder.header(...)` appends rather than replaces — without `.set` the inbound `Authorization` would shadow the injected one. The filter fails loud with `IllegalStateException` if the declared type has no registered injector, and emits DEBUG lines on both no-auth pass-through branches (`auth == null`, `auth.type == null`) so an operator can confirm "no Authorization header attached" is intentional.
    7. `removeRequestHeader("Cookie")` + `http()` forward.
- **Placeholder expansion via `Configuration.configureObject`.** `NativeAppParser` walks the typed POJO tree (`NativeAppFile` → `NativeAppConfig` → `Lifecycle`/`Security`/`Authentication`/`BasicAuthCredentials`/each `Command` and `ExposedPath`) and invokes `configureObject` on each owned node — same `${KEY}` / `${KEY}.{DEFAULT}` semantics as Jobs and other artefacts, just done in a recursive walk because `configureObject` only handles top-level fields. The typed config tree is **transient** on the entity; the raw post-expansion JSON is persisted in `NATIVE_APP_CONFIG_JSON` and rehydrated lazily by `NativeAppParser.rehydrateConfig` (used by the registry on cache miss, the synchronizer's lifecycle hooks, and the management endpoint).
- **JSON schema gotcha.** The user-facing `type` field (LOCAL/REMOTE) collides with `Artefact#type` (the artefact-type string) at the Gson layer (`Class declares multiple JSON fields named 'type'`). Solved by a dedicated DTO `NativeAppFile` for deserialization that mirrors the file structure 1:1; `NativeAppParser.parse` then maps the DTO onto the JPA entity. Don't try to deserialize into `NativeApp` directly.
- **Credentials field naming.** `Authentication.credentials.user` and `.password` are the literal username / password to send. Placeholders like `${SAMPLE_APP_USER}.{admin}` are resolved at parse time and persisted. The runtime injector reads the already-resolved values; no per-request env lookup. The management endpoint omits the credentials block from `GET` responses. `BasicAuthenticationInjector` throws `IllegalStateException` (surfacing as a proxy 500) when credentials are missing or `user` / `password` is null or blank — the message names the exact schema field path and adds a placeholder-resolution hint.
- **Management REST** at `/services/native-apps[/<id>][/start|/stop]` — list / get / start / stop / delete. Restricted by `HttpSecurityURIConfigurator.NATIVE_APPS_MANAGEMENT_PATTERNS` to **any of DEVELOPER, ADMINISTRATOR, OPERATOR** — distinct from the proxy itself which uses per-app scope semantics.
- **Integration-test patterns.** All live alongside other dirigible-fork ITs and complement each other:
  - `RemoteNativeAppIT` + `RemoteNativeAppTestProject` (`tests/ui/tests/...`, fixture under `tests/tests-integrations/src/main/resources/RemoteNativeAppIT/`) — `BaseTestProject`/`PredefinedProjectIT` pattern. Boots an in-JVM `HttpServer` for the upstream and uses ProjectUtil's `__UPSTREAM_URL__` build-time placeholder (deliberately NOT `${...}` to avoid colliding with the synchronizer's runtime placeholders). Exercises whitelist hit, whitelist miss, empty-base catch-all, basic-auth header injection, and delete-unregisters.
  - `LocalNativeAppLifecycleIT` (`tests/api/...`, fast HTTP-only) — spawns a tiny Java single-file source HTTP server via `java Server.java` (single-source execution since Java 11). Cross-platform, no Python or Node dependency. Verifies lazy/always start, monitor restart, basic-auth injection, clean shutdown. Two POSIX-only regression methods (`@EnabledOnOs(LINUX, MAC)`): `stop_command_subprocess_receives_DIRIGIBLE_NATIVE_APP_PORT` proves the env var reaches the stop subprocess; `stop_kills_descendant_processes` builds an `sh -c 'java Server.java; true'` chain (the `; true` suffix prevents bash's tail-call optimisation from `exec`ing into java — without it the held PID becomes java itself and there's no descendant to leak) and asserts every previously-snapshotted descendant PID is dead after `stop()`.
  - `RoleSynchronizerCleanupIT` (`tests/api/...`, HTTP-only) — covers the `.roles` artefact ↔ user-role-assignment cascade: publishes a `.roles` file → assigns the role to a fresh user → removes the `.roles` file → forces sync → asserts the role row is gone AND the user no longer carries the role. Was red before the FK cascade and goes green with it.
  - `SampleLibraryLocalNativeAppIT` (`tests/ui/tests/sample/...`, slow Selenide+Git-clone) — clones [`dirigiblelabs/sample-library-local-native-app`](https://github.com/dirigiblelabs/sample-library-local-native-app) (which holds `roles.roles`, `project.json`, and `sample-library-native-app-nodejs.native-app`), publishes via the IDE, creates a `library-reader` user, asserts 403, assigns `library-admin` via `SecurityUtil.assignRoleToUser`, asserts 200/201 for CRUD. Bumps `DirigibleConfig.NATIVE_APP_READY_TIMEOUT_MS` to 5 min to absorb the first-spawn `npm install`.
- **ITs do not need manual cleanup.** `IntegrationTest` already declares `@DirtiesContext(AFTER_EACH_TEST_METHOD)`, which destroys the Spring context between tests. That triggers `DirigibleCleanupBean.@PreDestroy` (drops H2 tables + deletes the dirigible folder) and `NativeAppShutdown.@PreDestroy` (stops every owned native-app process). Don't add `@AfterEach`-level cleanup that duplicates this — it just races the framework.
- **Gotchas worth remembering.**
  - `Python 3.14` on macOS has a bind regression in `http.server.HTTPServer` — don't use Python for embedded test servers; use `java Server.java`.
  - `ServerSocket(port, 0, InetAddress.getLoopbackAddress())` can succeed on loopback even when another listener is bound to `0.0.0.0:port` on macOS — this is why `PortResolver.isBindable` deliberately probes the wildcard interface (`new ServerSocket(port, 0, null)`). Keep it that way; reverting to a loopback probe will reintroduce silent `EADDRINUSE` failures when the parent platform (or any other process) is already on the preferred port.
  - `Spring Boot's DevTools` (or whatever `FileSystemWatcher` wires up) can deadlock against `sun.nio.fs.PollingWatchService$PollingWatchKey` on macOS during full IT runs (last seen on `JavaLspIT`, `CreateNewFileIT`, `JavaEntityDecoratorsSampleProjectIT`). This is NOT a native-apps regression. When the full reactor IT run hangs, jstack the surefire JVM and check for `Found one Java-level deadlock` between `main` and `FileSystemWatcher`.
